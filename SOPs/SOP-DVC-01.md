# SOP-DVC-01: Large Artifact Management

**Objective:** Prevent Git repository bloat, ensure massive datasets are reproducible, and implement a "Thin Client, Heavy Cloud" workflow.

## 1. The 100MB Rule
- **Mandate:** NO file larger than 100MB may ever be committed to Git.
- **Enforcement:** Ensure `*.h5ad`, `*.parquet`, `*.lance`, `*.npy`, and `*.pt` are explicitly added to `.gitignore`.

## 2. DVC Integration
- **Local Outputs:** Use `dvc add <path>` for files generated by local pipelines that need version control.
- **Cloud Outputs (The Data Lake):** Use `dvc import-url gs://<bucket>/<path>` to track remote artifacts without pulling their full mass to the local disk.

## 3. Tiered Data Catalog
Structure data according to these tiers:
- **Tier 0 (Raw Immutable):** Stored permanently in GCS. Never altered.
- **Tier 1 (Standardized):** LanceDB or Parquet files. Often tracked via `dvc import-url`.
- **Tier 2 (Experiment Results):** DVC-tracked outputs, grouped into folders named by `DEV-ID` and timestamp (e.g., `results/DEV26-001_20260415_120000/`).

## 4. Disk Safety
- Treat the local workspace as an ephemeral cache. 
- If a dataset is tracked via DVC and pushed to the remote, it is safe to `rm` locally to save space; it can always be restored via `dvc pull`.
